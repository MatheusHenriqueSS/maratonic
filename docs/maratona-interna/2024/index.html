Problem G. Pulos perdidos​

Dada uma sequência p de n (1≤n≤511 \le n \le 511≤n≤51) inteiros, queremos construir uma sequência a de q (n≤q≤51n \le q \le 51n≤q≤51) inteiros que satisfaça as seguintes condições:

  • a contém todos elementos de p.
  • o primeiro e o último elemento de a estão contidos em p.
  • a diferença absoluta (tamanho do pulo) entre o i-ésimo e o i + 1-ésimo elementos (posições do sapo) de a é 2i−12^{i-1}2i−1, para todo 1≤i<q1 \le i < q1≤i<q.

Se n=1n = 1n=1, não há pulos, portanto basta imprimir a sequência original. Então, vamos resolver para n≥2n \ge 2n≥2.

Primeiramente, podemos observar que os elementos em a devem ser distintos entre si. Isso porque cada pulo é uma potência de 2 distinta, e podemos provar que 2k>∑i=0k−12i2^k > \sum_{i=0}^{k-1} 2^i2k>∑i=0k−1​2i. Então, independente da direção do próximo pulo, não é possível o sapo voltar para uma posição anterior. Por exemplo, ainda que todos pulos tenham sido para a direita e o próximo seja para esquerda, este será maior que a soma de todos outros e portanto a próxima posição ficará à esquerda da posição inicial.

Além disso, como é garantido que sempre existe uma resposta para os casos gerados e a sequência da entrada p deve estar contida em a, temos que os elementos de p também sempre vão ser distintos.

Agora note que o primeiro elemento de a deve ser um elemento de p com paridade diferente de todos outros elementos em p. Isso se deve ao fato de que o primeiro pulo é o único com paridade ímpar, pois todos pulos são potências de 222, e 111 é a única potência de 2 ímpar. Dessa forma, ao dar o primeiro pulo, a paridade da posição do sapo muda, enquanto em todos outros pulos ela se mantém.

Com isso, para n≥3n \ge 3n≥3, o primeiro elemento de a pode ser unicamente determinado, sendo o único elemento com paridade diferente em p. Já para n=2n = 2n=2, os dois elementos de p podem ser o primeiro de a, e podemos escolher qualquer um deles para tal.

Assim, temos o primeiro elemento de a fixado, e vamos chamá-lo de s. Agora, podemos testar todas possibilidades de tamanho q (n≤q≤51n \le q \le 51n≤q≤51) e de último elemento da sequência a. Vamos chamar de t o elemento que estamos testando para ser o último elemento de a. Cada elemento de p diferente de s é um possível candidato para t.

Podemos observar que ao fixar q, s e t, há no máximo uma forma de sair de s e chegar a t com q−1q - 1q−1 pulos. Ela pode ser nossa resposta se, e somente se, ela contiver todos elementos de p. Então, para cada par (q,t), podemos gerar essa sequência e, se ela realmente for de s a t e contiver todos elementos de p, podemos imprimi-la como a resposta e finalizar a procura.

Agora veremos como gerar a sequência dados q, s e t. Vamos fazer o caminho ao contrário, saindo de t e chegando a s, começando com pulo de tamanho 2q−22^{q-2}2q−2 e dividindo o tamanho do pulo pela metade até chegar em 1. Usando novamente a observação de que 2k>∑i=0k−12i2^k > \sum_{i=0}^{k-1} 2^i2k>∑i=0k−1​2i, agora podemos concluir que se t−s>0t - s > 0t−s>0, o maior pulo deve ser para a esquerda, e se s−t>0s - t > 0s−t>0 (assuming correction from PDF text s - t < 0), ele deve ser para a direita. Isso porque o maior pulo é maior que a soma de todos os outros, então se ele for para uma direção que aumente a distância entre s e a posição atual, não será possível chegar a s com os pulos restantes. Então, só há chance de construir a sequência correta se o maior pulo diminuir a distância entre s e a posição atual. Isso resulta em um algoritmo recursivo, em que atualizamos t para a nova posição após o pulo e dividimos o tamanho do pulo pela metade, enquanto ele for maior ou igual a 1. No final, temos que reverter a sequência, pois fizemos o caminho ao contrário.

Dessa forma, ao fim do algoritmo de gerar a sequência dados q, s e t, temos todas posições em ordem da sequência candidata a a. Então só resta verificar se a sequência começa mesmo em s e contém todos elementos de p. Para verificar se ela contém todos elementos de p, pode-se armazenar em um set os elementos de p e para cada elemento da sequência candidata ver se ele está no set ou não. Se a quantidade de elementos contidos no set for igual a n, então a sequência candidata contém p, e pode ser a nossa sequência a.

A complexidade de tempo dessa solução é O(∣a∣2nlog⁡n)O(|a|^2 n \log n)O(∣a∣2nlogn), sendo |a| o tamanho da sequência da resposta, que no pior dos casos é 51. Mas vale lembrar que há t casos de teste independentes, portanto a complexidade de tempo final dessa solução considerando todos casos de teste é O(t∣a∣2nlog⁡n)O(t|a|^2 n \log n)O(t∣a∣2nlogn).

É importante lembrar que os valores da sequência p vão de −2⋅1018-2 \cdot 10^{18}−2⋅1018 a 2⋅10182 \cdot 10^{18}2⋅1018, portanto em C e C++, por exemplo, não é suficiente armazenar essas e outras variáveis em inteiros de 32 bits. Nesse caso, em C e C++, uma opção é usar o tipo long long int.

Problem H. Árvore de Strings​

Para cada vértice u, considere a string lexicograficamente maior s(u) que pode ser obtida pelo caminho de u pra algum vértice k em sua sub-árvore. Veja que k é folha, caso contrário, poderíamos descer a partir de k na árvore, adicionando assim caracteres no final de s(u), de modo que a obter uma string lexicograficamente maior, absurdo. Denote k por f(u), para todo vértice u da árvore. Vamos descobrir f(u) para todo u num approach bottom-up através de uma busca em profundiade começando na raiz.

O grande lance aqui vai ser implementar uma função bool compare(int a, int b, int c, int d), que recebe quatro vértices na árvore, sendo a ancestral de b e c ancestral de d, e retorna um booleano, sendo esse verdadeiro se a string obtida pelo caminho de a até b na árvore for lexicograficamente estritamente menor que a string de c até d, e falso caso contrário. Suponha que tal função está implementada por enquanto, e vejamos como resolver o problema com ela. Fala uma busca em profundidade começando em 1. Seja v o vétice atual. Atribua f(v)←vf(v) \leftarrow vf(v)←v e itere pelos filhos u de v. Execute a busca em profundidade em u, depois, se compare(v,f(v),v,f(u))\text{compare}(v, f(v), v, f(u))compare(v,f(v),v,f(u)) for verdadeiro, atribua f(v)←f(u)f(v) \leftarrow f(u)f(v)←f(u). Esse algorítmo eficientemente acha f(v) para cada vértice v em O(nα)O(n\alpha)O(nα), supondo que compare roda em O(α)O(\alpha)O(α). Podemos imprimir o tamanho do caminho de v até f(v) para cada vértice v, bastando pré-calcular a profundidade de cada nó na árvore (lembre que v é ancestral de f(v)). Por fim, para imprimir a string lexicograficamente maior de todas, basta inicializar variáveis a←1a \leftarrow 1a←1 e b←1b \leftarrow 1b←1 e, para cada vértice v, se compare(a,b,v,f(v))\text{compare}(a, b, v, f(v))compare(a,b,v,f(v)), atribuir a←va \leftarrow va←v e b←f(v)b \leftarrow f(v)b←f(v), de modo que resta apenas imprimir o caminho de a até b.

A questão é, como implementar compare(int a, int b, int c, int d)? Farei isso em O(log⁡(n))O(\log(n))O(log(n)) com busca binária e hash. Basta eu descobrir o maior inteiro não negativo k tal que os primeiros k caracteres do caminho de a até b coincidem com os do caminho de c até d, de modo que resta comparar os (k+1)-ésimos caracteres dos caminhos (e considerar os casos em que algum caminho tem tamanho exatamente k). Problem I. Desafio para o ChatGPT​

Como é um problema de MST, podemos pensar em usar um algoritmo de MST, porque não?

Os mais comuns são Prim e Kruskal, porém, se gerarmos todas as O(n2)O(n^2)O(n2) arestas já estaremos excedendo o tempo limite. Portanto, usaremos do Algoritmo de Boruvka. Ele consiste em:

Enquanto houver mais de uma componente conexa faça:

  • Para toda componente conexa CiC_iCi​ ache a aresta de menor custo w que liga a outra componente CjC_jCj​.
  • Ordene todas as arestas candidatas, e insira uma a uma, se isso não introduzir um ciclo, de forma similar ao algoritmo de Kruskal.

É possível notar que em cada etapa o número de componentes conexas divide em 2, pois o pior caso seriam termos arestas formando uma espécie de matching (agrupar em duplas).

Resta resolver o problema: como achar para cada componente a menor aresta que sai da componente. Uma opção que não entrarei em detalhes é com Centroid Decomposition, o que fica O(nlog⁡2n)O(n\log^2 n)O(nlog2n) e dificilmente irá passar, infelizmente.

A outra solução é com uma técnica de Re-root. Para cada vértice, basta manter os vértices de 2 cores (componentes) mais próximos. Assim, se um deles for a sua cor, o mais perto seria o outro. Esse é um problema clássico de re-root e pode ser feito em O(n)O(n)O(n) para complexidade final de O(nlog⁡n)O(n\log n)O(nlogn). Para mais detalhes dessa técnica veja:

  • Blog voltado mais a implementação: https://codeforces.com/blog/entry/124286
  • Vídeo em inglês: https://codeforces.com/blog/entry/96651
